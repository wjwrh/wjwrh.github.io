Game Hacking

一点写在前面的废话：

最近也算是有点迷茫了，陷入了一定程度的内耗，不知道自己到底应该做点什么，思来想去还是打算搞一搞Windows内核与游戏外挂，至于毕设具体要做什么项目..再说吧。

之前自己写过Apex的外挂，不过基本靠问@sad和缝合代码来实现的，自己对外挂还停留在一个缝合怪的阶段，对底层的一些知识可以说知之甚少，于是我到uncheat找到了一篇帖子：

[Discuss] How to get started in video game hacking? (My ultimate answer) (unknowncheats.me)

这篇帖子算是对我学习有所启发：

  • 不要眼高手低，一直保持学习的状态，相信质变一定会在某一天到来
  • 相信自己学的东西(cheat)一定是有用的，cheat包含相当多硬核的计算机知识，这对于安全从业者而言自然是十分有用的
  • 善于分解困难的问题，把困难的问题拆分成小问题，然后尝试去解决小问题，不要让一个大的困难的问题打击信心
  • 保持充足的激情学习，尽量多的动手，达成一些小挑战，提升自己的积极性
  • There is NO genius programmer.

然后这个帖子推荐了一本实践性比较强的书，也就是这篇博客存在的意义，这篇博客我同样不希望成为原书的抄录本，而是尽量作为一个备忘录，记载下来我在完成书上实验过程中碰到的问题，记录一下书上一些我认为有趣的东西，大概记录一下书上提供的使用方法与技巧。

我尽量在半个月到一个月的时间内读完这本书，然后去读推荐的其他书，相信自己可以做到吧，Believe myself

Part 1: Tools of the Trade

Chapter 1: Scanning Memory Using Cheat Engine

这一篇简要的介绍了一下CheatEngine的使用方法，CheatEngine可以帮助我们找到一些游戏中数据的位置，即Memory Scanner，然后可以通过Pointer Scanning来找到指向我们需要数据的基指针位置，最后就是也可以通过Lua引擎来实现一些复杂功能

这里稍微记录一下实验二，指针扫描

这里有几个选项值得注意一下

  • Stop traversing a path when a static has been found 这个选项，默认是没勾的，但是当你已经找到一个static指针的时候，基本上已经可以判定找到基地址了，所以通常情况这个可以勾上

  • Improve pointerscan with gathered heap data 如果程序没有自己定义内存分配器，用这个可以让CE通过操作系统堆分配器的记录信息，显著改善搜索结果(差不多就是，你offset的范围不太可能超出当前堆块的范围)，如果程序自己定义了内存分配器，则不要使用。通常是在第一次扫描时可以勾选此选项

  • Allow stack addresses of the first thread(s) to be handled as static 把第一个线程的堆栈地址视作静态地址

    用来应对这种情况

  • Maximum offset value && Max level 这两个可以从小往大调，逐步扩大搜索范围

Chapter 2: Debugging Games with OllyDbg

OD似乎已经太老了，于是我打算用x64dbg代替

对于Expressions书里给了两个例子，其中第二个用到了trace功能

一些基础功能我在之前自己整简单逆向的时候也算是体验过了，不过trace的功能，之前没有接触过

Shift+D打开x64dbg查看当前模块所有字符串引用

OllyDbg Expressions in Action

例1：当特定玩家名字被打印时暂停执行

这里基本上就是一个条件断点的使用，先通过CE等手段找到Player 1的名字在内存中存储的地址，然后下一个内存断点，(书里写的是发现正好断在他之前发现的printText函数之前，可能还需要结合一下其他分析方式)，然后在语句处下条件断点进行判断，就可以在打印Player 1名字之前断掉程序

例2：当玩家血量下降时暂停执行

这个例子的目的是找到在玩家血量从满血掉下来的时候执行的第一个函数，提供了两个思路

  • 在生命值处下断点，然后在每一个访问生命值的地方下条件断点，手动追踪每一个断点直到找到第一个函数调用

  • 使用OD的trace function去创建一个动态断点

    基本思路就是

    • 作者首先通过查找跨模块调用，发现大量对USER32.PeekMessageA的调用，这个函数一般是游戏窗口用于获取新Message使用的(最可能在main循环中被调用)

    • 通过简单分析发现这些函数调用确实存在一个大循环中，猜测这个就是主循环，在主循环中开始trace into，最后断在需要的位置

    • 启动trace，设置trace停止的条件(玩家血量低于max，且EIP的第一个字节为E8(call))

小练习：PATCHING AN IF() STATEMENT

这个我一开始有一个思路，就是球一定是有一个坐标(x, y)的，当坐标x超过某个值的时候，触发隐藏逻辑，因此如果可以找到x的地址找到所有访问x的地方，就可以找到判断x坐标的地方

通过CE，一开始搜索unknown value，然后勾选Pause the game while scanning，根据球的位置，选择Increased Value和Decreased Value进行多次扫描，最终锁定了唯一的一个四字节变量

在x64dbg中跳转到给定位置下断点，这里我们只需要断一个内存地址，而且是四个字节，可以下硬件断点(通过CPU寄存器功能实现)来增加效率

第一次断在了一个add函数里面

似乎是用来修改x坐标的，继续执行几次，就能找到将x坐标与某一常量比较，并根据比较结果跳转不同分支的逻辑

cmp是比较与140的大小，比140小则跳转，根据分析x小于140则是不隐藏，大于140则是隐藏

因此直接把jl给patch成jmp即可

patch之后小球不会隐藏

书上给的做法是在小球隐藏之后会出现字符串，通过这个字符串来定位关键函数，感觉比我的方法简单方便，说明搞逆向调试还是多注意一下程序的状态变化，特别是输出的交互内容等等。

Chapter 3: Reconnaissance with Process Monitor and Process Explorer

这一章主要介绍sysinternal的两个非常🐂的工具在对游戏分析过程中的应用

Process Monitor

常用快捷键表，也不一定用得上就是了((

procmon不仅可以在filter里面过滤一些比较细的东西，还可以在上方的菜单栏中选择是否输出某一大类的日志

procmon最强大的功能是可以追踪事件的调用栈

小练习：修改最高分文件

利用Procmon找到游戏存储最高分的文件，还是比较简单的

  • 设置ProcessName，仅打开文件操作监视
  • 过滤掉诸如C:\Windows之类的路径，可以发现游戏读取了%temp%\gh_hs_p.dat这个文件
  • 用010edior打开，发现一共四个字节，目前是全0，猜测应该是int32或uint32，先改全F(uint32 max)，打开游戏，发现最高分变-1了，应该是int32，改成int32.max就好了

Process Explorer

ProcessExplorer的Column可以选择ASLR Enabled查看某一进程的ASLR是否开启

ProcExp提供了String List，在游戏升级后可以diff两个版本的String，看看有没有新的有意思的String可以深入调研

小练习：关闭Mutex实现多开

ProcExp的实验非常简单，这里我想额外给自己加一点点难度，就是用Patch的方式实现多开，避免每次都要手动关闭Mutex

思路也非常简单

  • 通过ProcExp发现Mutex的名字为Chapter3_CloseMutex-Mutex，按Shift + D打开当前模块字符串表，找到并且直接跟到引用这个字符串的地方
  • 发现了CreateMutexA和GetLastError以及输出Only one instance...的调用，判定这里就是判断逻辑，直接改跳转语句即可
  • 成功实现多开

Chapter 4: From Code to Memory: A General Primer

这一章主要讲C++的变量在内存中的分布情况，一般不足4byte的变量会对齐到4byte，结构体中的成员对其会根据编译器的设置进行对齐

另外就是一些VFT的知识，如果某些类由抽象类派生而来，那么在内存中该类的第一个成员会变成该类的虚表

段寄存器

在Windows x86实现中，通常

DS(数据) = SS(stack) = GS = ES

DS - CS = 8

FS - DS != 0

FS会指向与当前线程相关的某些数据，在bypass ASLR的时候可能会用到

DIV&&MUL

MUL(signed)和IMUL(unsigned)均可以仅接受一个操作数，根据操作数的大小可能分别会与EAX AX AL相乘，并且把结果存在大一级的寄存器中

Chapter 7: Code Injection

shellcode注入

这章主要介绍了写shellcode然后注入到目标执行的方案，书中给出了两种方案，两种方法的共同之处在于都是通过VirtualAllocEx申请内存，然后WriteProcessMemory将shellcode以及参数写入内存(DWORD之类的参数可以直接硬编码进SHELLCODE，字符串之类的要写到分配的内存中，将指针替换掉shellcode的相关部分)

  • 新建线程执行shellcode

    • CreateRemoteThread

  • 劫持现有线程执行shellcode

    • SuspendThread

    • GetThreadContext

    • 记录原EIP，修改Context.EIP指向shellcode

    • SetThreadContext

    • 将原EIP写入shellcode中

    • ResumeThread

第二种方式可以与游戏主进程同步执行一些代码

DLL注入

书中通过GetModuleHandle + GetProcAddress加载LoadLibraryW，通过CreateRemoteThread在目标进程创建新线程调用LoadLibraryW加载DLL

注意 DLLMAIN是在获取loader lock的情况下被调用的，这个lock用于保护进程的模块列表，所以尽量不要在DLLMAIN中执行太多操作，以避免死锁

可以在其中创建新线程进行其它操作

在注入线程后，可以直接读取TEB(FS指向它)，TEB + 0x30指向PEB，PEB + 0x8即存储基地址，可以这样绕过ASLR获取基址

NOP实验

这里这个实验程序我运行的时候出现了一点小问题，搜索NOP点的代码无法找到合适的NOP点

打开IDA分析后，发现VS编译出来的结果很奇怪，给定函数的地址处不是实际的函数逻辑，而是一段JMP跳转，跳转过后才是真正的函数逻辑

查了一下这是VS的增量链接机制导致的，该机制通过这种方式，可以在对部分函数进行修改后不重构整个文件(MSDN也没讲太清楚，不过个人理解是，如果某个函数修改之后体积过大不能放在原先的位置，可以开辟一个新空间放下来，然后只用修改这里jmp的地址就行)，可以显著缩短编译耗时

关掉这个选项就正常了

Call Hooking

near call的操作数是相对地址，相对于call后一条指令地址的相对地址

写替换的新函数时，要注意调用约定、返回类型、参数个数(其实个人感觉在cdecl中这个不一定要一样)要相同

这里是VS插入的对JMC功能的实现代码，关闭这个功能就好

此法需要修改代码段，较易被检测

VF Table Hooking

编译器在对某个调用含糊不清时才会从虚表进行调用；最常见的是通过指针调用inst->SomeFunc()，通过引用的调用可能也会触发虚表调用

个人理解：通常基于现有游戏引擎制作的游戏都会通过虚表调用(因为往往他们需要继承游戏中的一些基类，然后基类中会预定义一些代码，这些代码里使用的是指针调用以实现动态绑定)

虚表中函数的调用方式为__thiscall，即通过ECX转递this指针，可以通过内联汇编的形式快速声明这样的函数

IAT Hooking

IAT位于PE头部，在加载模块后，其所有依赖的模块都会被递归加载，载入所有模块后，所有模块的IAT将会被解析，填进导入函数的真实地址

PE头部格式比较复杂，首先是DOS头，随后才是PE头

PE头的DataDirectory[IMAGE_DIRECTORY_ENTRY_IMPORT].VirtualAddress存放了指向IMAGE_IMPORT_DESCRIPTOR结构体数组第一个元素的RVA

每一个结构对应一个依赖模块，这个结构中又有两个数组，这两个数组一个存储了函数名称，另一个存储了函数地址

具体遍历代码比较复杂，参考原书

Jump Hooking

Jump Hooking直接在程序指定位置将原有指令替换为jmp指令，jmp大小为五个字节，这可能会覆盖一些其它关键指令，因此我们不能直接jmp到功能函数，我们需要跳转到trampoline，保存上下文，完成调用，并且执行被我们覆盖掉的字节，然后跳转回之后的指令，以下是示意图

trampoline函数的生成需要在前声明__declspec(naked)来组织编译器自动生成函数序言等指令，避免破坏堆栈
