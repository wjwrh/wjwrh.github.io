这里有几个选项值得注意一下

  • Stop traversing a path when a static has been found 这个选项，默认是没勾的，但是当你已经找到一个static指针的时候，基本上已经可以判定找到基地址了，所以通常情况这个可以勾上

  • Improve pointerscan with gathered heap data 如果程序没有自己定义内存分配器，用这个可以让CE通过操作系统堆分配器的记录信息，显著改善搜索结果(差不多就是，你offset的范围不太可能超出当前堆块的范围)，如果程序自己定义了内存分配器，则不要使用。通常是在第一次扫描时可以勾选此选项

  • Allow stack addresses of the first thread(s) to be handled as static 把第一个线程的堆栈地址视作静态地址

    用来应对这种情况

  • Maximum offset value && Max level 这两个可以从小往大调，逐步扩大搜索范围

VF Table Hooking

编译器在对某个调用含糊不清时才会从虚表进行调用；最常见的是通过指针调用inst->SomeFunc()，通过引用的调用可能也会触发虚表调用

个人理解：通常基于现有游戏引擎制作的游戏都会通过虚表调用(因为往往他们需要继承游戏中的一些基类，然后基类中会预定义一些代码，这些代码里使用的是指针调用以实现动态绑定)

虚表中函数的调用方式为__thiscall，即通过ECX转递this指针，可以通过内联汇编的形式快速声明这样的函数

IAT Hooking

IAT位于PE头部，在加载模块后，其所有依赖的模块都会被递归加载，载入所有模块后，所有模块的IAT将会被解析，填进导入函数的真实地址

PE头部格式比较复杂，首先是DOS头，随后才是PE头

PE头的DataDirectory[IMAGE_DIRECTORY_ENTRY_IMPORT].VirtualAddress存放了指向IMAGE_IMPORT_DESCRIPTOR结构体数组第一个元素的RVA

每一个结构对应一个依赖模块，这个结构中又有两个数组，这两个数组一个存储了函数名称，另一个存储了函数地址

具体遍历代码比较复杂，参考原书

Jump Hooking

Jump Hooking直接在程序指定位置将原有指令替换为jmp指令，jmp大小为五个字节，这可能会覆盖一些其它关键指令，因此我们不能直接jmp到功能函数，我们需要跳转到trampoline，保存上下文，完成调用，并且执行被我们覆盖掉的字节，然后跳转回之后的指令，以下是示意图

trampoline函数的生成需要在前声明__declspec(naked)来阻止编译器自动生成函数序言等指令，避免破坏堆栈

在Direct3D上应用Jump Hook和虚表Hook

D3D的整体代码框架大致如下图所示

drawFrame中，代码会先后调用device对象的以下方法

  • Clear 清理屏幕
  • BeginScene 解锁场景进行绘图
  • 调用一些绘制函数
  • EndScene 锁定场景
  • Present 将场景渲染到屏幕上

但我们就算注入到目标进程中，调用相同的D3D API获取到一个新的device实例，该实例与游戏中原有device的虚表也不是同一张

D3D设备使用一个自定义运行时实现的虚表，每一个设备有自己独立的虚表，并且设备有时会重写他们的虚表移除hook，因此对D3D的hook只能找到游戏原本的device直接对其虚表进行hook

但由于大部分device并不会重写EndScene方法，因此对D3D设备的HOOK可分为以下步骤

  1. 利用D3DAPI创建新的device

  2. 读取新device的虚表，定位到EndScene函数

    DIRECT3DDEVICE9的基类声明是公开的，可以在d3d9.h中找到，可以轻松地发现EndScene在虚表中的序号为42

  3. 在EndScene函数中下Jump Hook，获取目标device后解hook，跳转回原EndScene执行

    需要注意的是d3dx9使用的是C语言规范，EndScene的调用约定并不是__thiscall，而是通过栈传递对象指针

  4. 获取目标device后即可对目标虚表开展hook

D3D device有时会恢复VF table，这通常发生在EndScene中，因此在我们hook EndScene，执行原EndScene后需要再埋一次hook

书中第八章给的示例代码有一点点潜在的问题，即在埋JMP Hook的时候，如果游戏主线程正在执行EndScene的前5个字节对应的指令，会出问题，有两个解决方法

  • 对PeekMessage进行IAT hook，在IAT hook中对EndScene进行hook，这种有一个小问题，就是必须确保PeekMessage位于游戏主线程中
  • 一种更安全的方法是，挂起游戏主线程，同时确保主线程EIP不在EndScene头五个字节中，随后恢复线程执行

Z-Buffering

这是一种实现将重叠的物品正确的绘制到屏幕上的一种技术

实现一个二维数组，数组的大小等于绘制出二位画面的大小，数组存储着Z变量，即”这个物体距离屏幕的距离”

Z值越小的将优先绘制，覆盖Z值较大的物体

D3D9通常使用device->DrawIndexedPrimitive来绘制图形，因此可以hook该函数，在检测到特定传入参数时，关闭Z-Buf功能，调用绘制函数，再开启Z-Buf

这样可以保证我们的特定图形一定会被绘制出来，同时我们还可以更改图形的材质属性

Chapter 12: 保持隐蔽

这一章主要介绍了几个反作弊软件，以及一些绕过他们的技巧

常见AntiCheat软件

  • PunkBuster

    • 内存SBD(Signature-Based Detection) 扫内存，检测匹配的pattern(很容易误报)，主要是通过若干隐藏线程在用户态调用NtQueryVirtualMemory实现
    • 截图
    • 检验游戏关键文件hash(只针对在硬盘上的文件)

  • ESEA Anti-Cheat

    手段与PunkBuster类似，不过很多操作在内核态中完成

    • MmGetPhysicalMemoryRanges
    • ZwOpenSection
    • ZwMapViewOfSection

  • VAC

    手段也基本相似，不过VAC会进行DNS缓存扫描，查看主机是否访问了作弊软件相关的网站

    除此之外VAC还会进行二进制文件验证，它会对内存中游戏的关键字段算hash进行比对

  • GameGuard

    同样使用SBD，并且它同时拥有用户态和内核态功能，hook一系列外挂可能利用的函数

  • Warden

    动态下发代码，这段代码会执行检测任务并且定期向服务器发送心跳包以确保反作弊正在正常运行

小心地管理Bot留下的足迹

这里大致的说了几点常规思路，Hook尽量少的函数、可以选择Hook Windows API而不是游戏内的函数，给程序加壳，增加反调试功能

同时书里给出了若干可用的反调试手段

  • CheckRemoteDebuggerPresent 非常常见也非常容易绕过的反调手段
  • 通过SEH执行INT 0x2D INT 0x03判断是否有调试器挂载
  • 检测DR0-DR3
  • 检测OutputDebugString是否会执行出错
  • 调用时间相关API判断某个操作是否消耗异常多的时间
  • 检测是否有调试驱动存在

对抗反作弊

  • 对抗SBD

    仅加壳并不是一个好主意，可以考虑hook掉NtQueryVirtualMemory之类获取内存的函数

    或者考虑修改一些编译选项，以产生具有不同特性的二进制文件，不过如果需要大量分发外挂，这也不是一个长久之计

  • 对抗截图

    可以直接hook用于截图的函数；也可以hook在截图之前一定会调用的某个函数，在该函数内短暂的隐藏ui

  • 对抗二进制文件校验

    尽量hook Windows API 而不是游戏文件；同样也可以Hook内存查询函数来欺骗反作弊

  • 对抗反作弊的Rootkit

    有些反作弊会hook用户态最底层的ntdll.dll中的系统调用函数，例如WriteProcessMemory，可以检测到对游戏进程发起的内存写入，我们可以直接copy一个ntdll.dll，载入新的拷贝再执行

后记

这本书也介绍了不少写外挂的思路吧，尽管我感觉跟目前主流反作弊对抗还有一些差距，不过至少有了一些游戏作弊的设计思路，之后可能会考虑对AssaultCube这种没有反作弊保护的游戏写一下外挂试试
