2. Programming a Guessing Game

本章节以一个猜数字程序为例介绍了各个方面的基础知识点，内容比较杂比较浅显

变量和引用

Rust中使用let关键字创建变量，默认情况下Rust的变量是不可变的，想要创建可变变量需要添加啊关键字mut

在给函数传参时使用&符号即可传递引用，默认情况下引用也是不可变的，需要使用mut关键字获取可变引用

疑问：如果以可变引用引用不可变变量会如何？
答：很明显会报错。。

基本错误处理

read_line会返回一个Result类型的值，Result值可能为Ok或者Err
如果想在遇到Err时不做处理简单的终止程序，可以调用Result的方法expect
如果对于一个可能出错的Result不做任何处理，编译器会发出警告

依赖引入与依赖版本

Cargo理解语义化版本规范，在0.x.y的开发阶段，x相同但y不同的依赖理应可以共用，因此cargo会使用x相同但y尽量大的依赖库
例如当前最新版本的rand为0.8.5，使用rand = "0.8.3"会引入0.8.5版本的rand

为了保证完全一致的特性，Cargo使用Cargo.lock来确保第一次build之后的所有build使用相同版本的依赖，例如

  • 在初始时cargo.lock为空，在编译时rand最新版本为0.8.5，配置文件指定版本0.8.3，则首次执行cargo build会拉取0.8.5版本的rand，cargo会将rand及其依赖的相关信息写入lock文件
  • 后续出现了版本0.8.6后，重新build此项目，cargo会检查lock文件，发现此项目首次编译使用0.8.5，便不会采用0.8.6版本，而是使用0.8.5版本编译
  • 如果需要强制使用最新修订号的依赖，需要使用cargo update指令

  • 可以对const执行可变引用，但是引用指向const的副本，不会影响原有值；对普通不变量进行可变引用无法通过编译

debug模式下整型溢出会导致panic，release不会

4. Understanding Owwnership

Ways Variables and Data Interact

Move

对于基本数据类型的传递，Rust与其他语言类似，直接复制值

x与y是两个独立的变量，指向栈上不同的空间

而对于引用类型的传递，Rust采用move的形势实现

在其它语言中，s2与s1内部的指针会指向同一片内存区域，而在Rust中，s1会直接失效，只有s2拥有该片内存区域的所有权

Clone

对于实现了Copy trait的类型（基本数据类型实现了），在复制时会自动进行深复制，同时复制前后的变量都不会失效

如果想让没有实现copy的引用类型执行深拷贝，需要调用clone方法

需要注意的是，函数参数的传递以及参数的返回都有类似的行为，实现了Copy的进行深复制，没有实现的会发生Move

References and Borrowing

为了简化代码，避免传参等情况导致的owner权限丢失，可以传递参数的引用

函数参数必须要声明其接受引用，调用时也必须使用&符号获取引用并传递

如果原变量是可变的，那么还可以获取其可变引用，但是如果同一个scope中已经有一个可变引用出现，则该变量不能再有其他的引用

准确来说是，如果一次可变借用发生了，那么在此次可变借用发生之前的任何借用不能再使用

这种借用是可以发生的，但是在mut_ref2之后就不能再使用inmut_ref和mut_ref1了

此外Rust编译器确保不会出现悬垂引用的情况

The Slice Type

String slice是一种对于某个String的特殊的引用，其类型为&str

7. Managing Growing Projects with Packages, Crates, and Modules

Packages & Crates

crate是Rust编译器处理代码时的最小单位，crate分为binary crate和library crate

binary crate可以编译为可运行程序，library crate可以编译成库

package是一个或多个提供一系列功能的组合，一个package可以包含任意个bin crate，但只能包含一个lib crate

src/main.rs 是crate root

[rust api guidelines](About - Rust API Guidelines (rust-lang.github.io))

Modules

crate root可以创建新的模块，在非根模块中也可以创建子模块

Best Practices for Packages with a Binary and a Library

通常使用use希望将某个函数引入scope时，不直接将其引入，而是引入其父模块，这样能够更好的区分从外引入的对象与本地对象；而引入结构体、枚举时则通常直接引入，这是一种习惯

Windows Internal 随记

发表于 2022-07-12 更新于 2022-09-05 分类于 OS

Windows Internal

CHAPTER 2 System architecture

Hypervisor

VTL 0与VTL 1是一种类似隔离的机制，与原有的Ring机制有一定的独立性，VTL 0的内核无法访问VTL 1的用户态内存，虚拟机监视器是操作系统第一个初始化的组件，他会设置I/O MMU（避免通过DMA直接访问物理内存）以及SLAT，用来控制VTL 0中的内存访问。

VTL 1的secure kernel使用独立的二进制文件securekernel.exe，同时IUM的dll也有所不同，lumdll.dll对应ntdll.dll，lumbase.dll对应Kernelbase.dll

User Mode

subsystem

子系统有以下主要构件

  • 一个环境子系统进程实例Csrss.exe以及其加载的四个DLL，Basesrv.dll，Winsrv.dll，Sxssrv.dll，Csrsrv.dll
  • 位于内核态的Win32k.sys
  • 为控制台提供支持的Conhost.exe
  • 将多个可视窗口合成渲染到一个图形界面上的dwm.exe
  • 子系统DLL
  • 硬件的图形化设备驱动

现版本Windows仅有一个subsystem，名称也同为windows，每一个Session都有一个进程名为Csrss.exe，该进程为环境子系统运行实例，子系统之上的应用程序调用特定API时会出现以下三种情况

  • 该API实现功能较为简单，直接在特定dll内部即可完成
  • 需要切换到内核态实现功能，此时API代码会调用ntdll.dll内的桩代码进入内核态
  • 需要在子系统进程(Csrss.exe，Client/Server Runtime SubSystem)完成，此时subsystem dll通过ALPC向子系统进程发出请求，同时等待其回应

在Win8之后，cmd.exe会创建conhost.exe线程，后者用于与ConDrv.sys通信，该驱动用来处理I/O输入，用来减轻Csrss.exe上下文切换的压力

WSL

WSL(Windows Subsystem for Linux)与上述environment subsystem不隶属同一范畴，WSL使用Pico Provider机制，使用LXCore与LXSS两个驱动，向内核注册自身，在此之后便可由这两个驱动处理用户态Pico Process的各项系统调用

为了更好地实现Linux的特性，适配Linux程序，Windows与Linux之间有一部分类似但具有一定差异的特性（例如具名管道）不能重复利用，只能重新实现。

ntdll.dll

包含两类代码

  • 系统调用桩代码
  • 被subsystem、subsystem dl以及其它native镜像使用的内部函数
    • 以Ldr开头的image loader
    • 堆管理器
    • 以Csr开头的子系统进程通信函数
    • 以Rtl开头的运行时库例程
    • 以DbgUi开头的用户态调试支持函数
    • 以Etw开头的Windows事件追踪(Event Tracing for Windows)函数
    • 用户态异步过程调用调度器以及异常调度器

桩代码会检查 SharedUserData + 0x308 字段决定使用syscall还是int 2Eh进入系统调用，当启用了Credential Guard VBS属性时，会使用int 2Eh，因为相较于syscall，hypervisor对int指令处理更有效率

Native images

部分进程运行会早于子系统初始化，这些进程仅依赖ntdll.dll，例如Smss.exe(Session Manager process)，Smss是第一个用户态进程，由内核直接创建，同时由他来负责启动Csrss.exe

Kernel Mode

Executive

是Ntoskrnl.exe的上层部分，其中包含了下列种类的函数

  • 对外导出且可在用户态调用的函数 这些函数被称为system service并且通过Ntdll.dll导出，例如NtCreateFile。这些函数绝大多数都可以通过Windows API或者子系统API访问，也有一少部分无法通过记录的API调用。
  • 通过DeviceIoControl调用的设备驱动函数 它们提供了一个用户模式到内核模式的通用接口，用于调用设备驱动中与读写无关的函数(?)
  • 记录在WDK中只能在内核态调用的导出函数
    • 以Io开头的I/O管理器例程
    • 以Ex开头的通用executive函数
  • 已被导出但未记录在WDK中能在内核态调用的函数 包含了由boot video driver调用的函数，以Inbv开头
  • 并未导出但是被定义为全局符号的函数 这些函数包含内部支持函数，例如以Iop开头的内部I/O管理支持函数以及以Mi开头的内部内存管理支持函数
  • 在模块内未被定义为全局符号的函数 这些函数只能由executive和kernel使用(即只能在ntoskrnl内使用)

该层内有以下主要组件

  • Configuration manager 配置管理器，负责实现与管理注册表
  • Process manager 进程管理器，负责创建与终止进程和线程，进程和进程的底层实现由下层的内核提供，该部件为这些底层对象提供更多的语义及函数(个人理解即进行了包装，提供了更多的操作)
  • Security Reference Monitor SRM，安全参考监视器，强化本机的安全政策，保护系统资源，执行运行时对象审查与保护
  • I/O manager 实现独立于设备之上的I/O操作，并且将I/O发派给合适的设备驱动做进一步处理
  • Plug and Play(PnP) manager PnP管理器，判断某一设备需要哪些驱动，并且为该设备加载这些驱动。它会在枚举时查询每一个设备的硬件资源需求，根据这些需求来为设备分配硬件资源，例如I/O ports, IRQs, DMA以及memory location。它同时也负责为系统上的设备更改发送合适的事件通知。
  • Power manager 电源管理器，processor power management(PPM)以及power management framework(PoFx)协调电源事件同时向设备驱动生成电源管理I/O通知。
  • Windows Driver Model (WDM) Windows Management Instrumentation (WMI) routines 这些例程允许设备驱动发送性能和配置信息，同时允许从用户态WMI服务接受命令
  • Memory manager 内存管理器实现了虚拟内存，同时为cache manager提供底层支持
  • Cache manager 缓存管理器通过多种手段改善基于文件的I/O性能。将最近引用的磁盘数据暂存在主内存中，将数据的改动先留存在内存中并之后在合适时写入内存

其中有四大类支持函数

  • Object manager 创建、管理、删除Windows executive对象
  • Asynchronous LPC (ALPC) facility ALPC设施用于在同一台计算机上的客户进程和服务进程之间传递信息
  • Run-time library functions 包括字符串处理、算术操作、数据类型转化以及安全结构处理
  • Executive support routines 包括了内存分配、互锁内存访问以及一些特别的同步机制

Kernel

由ntoskrnl中提供基础机制的一系列函数组成，其中也有相当一部分对于驱动开发而言必须的函数记录在WDK中，这些函数以Ke开头

Kernel Object

内核为上层的executive提供底层支持，同时内核只实现OS的基本机制，避免制定执行策略（除了线程调度）

executive层将诸如线程一样可共享资源视作对象，且提供了大量的机制以便利用对象，而kernel层中的对象没有这些机制，仅仅用来帮助内核控制处理器以及为上层提供支持，大多数executive对象由一或多个内核对象封装而来

  • control objects 控制各种操作系统功能
    • Asynchronous Procedure Call (APC)
    • Deferred Procedure Call (DPC)
    • 一些I/O管理器会用到的对象，例如中断对象
  • dispatcher objects 包含内核线程，互斥锁，事件，内核事件对，信号量，计时器和可等待计时器
Kernel processor control region and control block

内核利用kernel processor control region(KPCR)来记录一些处理器信息，例如IDT, task state segment(TSS), GDT

为了便于处理器访问KPCR数据，处理器存储KPCR指针到fs寄存器(32位系统)或gs寄存器(64位系统)中

KPCR是对外公开的，可以被其他驱动和系统组件使用，而KPCR内有一个内嵌结构kernel processor control block(KPRCB)，该结构只被ntoskrnl.exe中的内核代码使用

经过实验，每一个逻辑核心都有自己独立的KPCR和KPRCB(6核12线程实际上是12个逻辑核心，这里用到了超线程技术)

Part 1 P78

内核的另一项主要工作是将executive和设备驱动程序从Windows支持的硬件架构之间的变化中抽象、隔离出来

Hardware abstraction layer

HAL对硬件访问进行了抽象，为依赖于特定架构或依赖于特定机器的功能提供了统一的接口，例如

  • I/O interface
  • interrupt controllers
  • multiprocessor communication mechanisms

通过HAL提供的例程访问硬件可以保证代码的可移植性

x86架构下提供两个HAL文件，Halacpi.dll和Halmacpi.dll，在Windows启动时会自行决定应当使用哪一个

HAL File Name Systems Supported
Halacpi.dll Advanced Configuration and Power Interface (ACPI) PCs. 只有一个核心且不支持APIC
Halmacpi.dll Advanced Programmable Interrupt Controller (APIC) PCs with an ACPI. 支持APIC意味着支持SMP

x64和arm架构下因为处理器需要ACPI和APIC支持，所以只有一个HAL

Windows使用HAL扩展模块对不同硬件提供支持，例如Intel DMA controller可能不会出现在某个特定的设备上，或者有时需要对不同的controller提供支持，

Device drivers

设备驱动是连接I/O管理器与相关硬件的可载入内核模块，他们在内核中会以一下三种上下文之一运行

  • 用户线程发起调用了I/O函数
  • 内核模式系统线程(例如来自PnP管理器的请求)
  • 由于中断的出现，从而不在某一具体的线程上下文中

Windows设备驱动并不直接操作硬件，而是通过调用HAL内的函数与硬件交互

设备驱动分为以下几类

  • Hardware device drivers 利用HAL操控硬件来向物理硬件或网络写出输出或者查询输入
  • File system drivers 接受面向文件的I/O请求并把它们翻译为与特定设备关联的I/O请求
  • File system filter drivers 执行磁盘镜像和加密，或扫描定位病毒程序，拦截I/O请求
  • Network redirectors and servers 这是将文件系统I/O请求传输到网络上的计算机中的文件系统驱动
  • Protocol drivers 实现网络协议
  • Kernel streaming filter drivers 这些驱动链接在一起用来在数据流上进行信号处理
  • Software drivers 这些内核驱动为某些用户模式进程执行一些只能在内核模式下完成的操作
Windows driver model

WDM将驱动分为三大类

  • Bus drivers 总线驱动为总线控制器、适配器、桥或者任何拥有子设备的设备提供服务，总线驱动是必备的驱动，每一种总线都有一个总线驱动
  • Function drivers 功能驱动是设备驱动的主驱动，并且为设备提供可操作界面
  • Filter drivers 过滤驱动程序为一个设备或现存的驱动增加更多的功能，或者去修改其它驱动发出的请求以及收到的响应

Windows Driver Foundation (WDF) 提供了KMDF和UMDF来简化驱动开发

Routine Commonly Used Prefixes

Windows内核组件导出函数的前缀都是有规律可循的，具体常用前缀表见Part 1 p87

同时这些组件也会使用这些前缀的变体来标识内部函数，例如用前缀的第一个字母加i(internal)或者用前缀的全拼后加p(private)，例如

  • Ke指Kernel，而Ki指内部内核函数
  • Ps指Process support，Psp指内部进程支持函数

函数的命名方式为<Prefix><Operation><Object>，例如ExAllocatePoolWithTag

System processes

System idle process

该进程没有对应的用户态可执行文件，进程ID为0

这个进程内线程的数量与逻辑CPU的数目相等，该进程用于统计空闲时间

System process and system threads

这里的System process指PID为4名称为System的进程

该进程是所有内核模式线程(一种仅运行在内核态的线程)的家，系统线程拥有所有常规用户态线程拥有的属性和上下文，例如硬件上下文、优先级等，但不同的是系统线程仅运行内核模式的代码，且没有用户模式内存空间，因此只能从OS内存堆中分配动态内存

系统线程通过PsCreateSystemThread或者IoCreateSystemThread函数创建，Windows或其他设备驱动在系统初始化时创建系统线程来执行需要线程上下文的操作，例如发出或等待I/O及其它对象，或者进行设备轮询

* Memory Compression process

内存压缩进程使用其用户模式地址空间来存储与已从某些进程的工作集中逐出的备用内存相对应的压缩内存页(机翻没太看明白，看原文也没太看明白，可能到时候回来再看看)

Session Manager

执行kernel和executive初始化最终阶段的内核模式系统线程创建了这个进程

主Smss.exe会查询大量注册表信息完成一系列初始化操作(见Part1 P92)，在最后会创建子Smss.exe实例来初始化session 0以及session 1

子Smss.exe在完成初始化之后会退出执行，只有主进程会保持执行

Windows initialization process

同样也完成了大量初始化操作，创建了一些事件对象，设置了一些基本变量，初始化了service control manager

Service Control Manager

Windows下的服务类似于Linux的守护进程，可以设置他们在Windows启动时自动启动

SCM是一个特殊的系统进程，它负责启动、停止以及与服务进程交互；服务程序实际上也只是会调用Windows函数与SCM交互的特殊的Windows image

服务与服务进程并非一对一关系，部分服务会共享一个进程

Winlogon, LogonUI, and Userinit

Winlogon由初始化session的Smss.exe调用，它负责与用户交互实现登录和登出，当用户按下secure attention sequence (SAS) 键盘组合时，Winlogon就会接收到这个信息并且进入登陆界面，默认SAS组合是Ctrl+Alt+Del，这个组合不能被用户态应用拦截，可以一定程度上保障用户登录安全

当用户成功登录后，Lsass.exe会为用户生成合适的访问凭证对象，如果用户是管理员用户且系统启用了UAC，那么还会额外生成一个受限的凭证；随后Winlogon会使用凭证来创建用户会话初始化进程，默认是Userinit.exe，Userinit.exe会在初始化一些环境后，调用HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon\Shell中存储的程序(默认是explorer.exe)

CHAPTER 3 Processes and jobs

Creating a process

创建进程的Windows API

  • CreateProcess 创建与调用进程拥有相同访问令牌的进程
  • CreateProcessAsUser 使用额外的访问令牌创建进程
  • CreateProcessWithTokenW
  • CreateProcessWithLogonW

后面两个API均来自advapi32.dll，CreateProcessWithTokenW和CreateProcessAsUser类似，均接受一个HANDLE作为用户Token，并以该用户的身份创建进程，二者的差别是对主调进程的权限要求不同，二者会通过发出RPC(remote procedure call)请求调用Secondary Logon服务

上述四个例程都需要一个可运行文件来创建进程，Windows还提供了ShellExecute(Ex)来针对特定的文件执行”open”操作，Windows会查找注册表中的扩展名关联信息，启动合适的文件，最终也会调用CreateProcess来创建进程

所有的进程创建最终都会调用ntdll.dll里的NtCreateUserProcess切换到内核态完成内核中的用户进程创建工作

其他任何形式的进程的创建，包括UWP、native process、pico process以及一些系统进程(没有具体的镜像的进程, minimal)这些，最终都会在用户态调用NtCreateUserProcess或NtCreateUserProcessEx，而这两者虽然是不同的系统调用，但他们在进入内核态之后却使用了同一个例程来完成内核态的工作

Process internals

每一个进程都会被一个EPROCESS结构体所代表，该进程包含了许多进程相关的属性以及指向与进程相关结构体的指针

例如EPROCESS::Win32PROCESS，这是一个指向W32PROCESS的指针，后者是Win32k.sys用于维护windows subsystem的结构体，需要注意的是在非Windows Subsystem进程(例如native process，smss.exe)的EPROCESS内该指针值为null

内核驱动可以使用PsSetCreateProcessNotifyRoutine(Ex, Ex2)例程来在进程创建时执行一些操作(例如创建一个用于记录进程信息的结构体)

EPROCESS的第一个成员Process Control Block(PCB)是KPROCESS的结构体对象，kernel内核层会使用该对象来完成一系列操作，将EPROCESS和KPROCESS区分开是为了保持一定程度上的抽象

EPROCESS中有一个Active Process Link，它是一个由nt!PsActiveProcessHead为表头组成的双向链表，链接起了系统中所有的活跃进程，需要注意的是链表中的FLINK和BLINK指向的是临近节点中对应的Active Process Link，而非EPROCESS

经过自己实验，nt!PsActiveProcessHead实际上指向System(PID为4)的EPROCESS的Active Process Link结构

EPROCESS以及进程相关的结构体大都存放在内核空间中，但是有一个例外，Process Environment Block(PEB)存放于用户空间中，EPROCESS中的PEB成员指向该结构体(system:4的EPROCESS.Peb为NULL)

Protected process

正常情况下，一个拥有调试权限的进程可以完全访问此计算机上的任意其他进程，但为了能在迎合数字版权要求的前提下在Windows上播放蓝光之类的媒体影片，Windows推出了protected process

只有获取了特殊签名的应用程序才能被创建为保护进程，保护进程的EPROCESS中会设置一个特殊的位，这会导致一些安全相关的进程拒绝提供保护进程的部分权限；实际上对于保护进程能够获取的权限仅有

  • PROCESS_QUERY/SET_LIMITED_INFOMATION
  • PROCESS_TERMINATE
  • PROCESS_SUSPEND_RESUME

Protected Process Light (PPL)

PPL机制位进程保护添加了一种等级关系，不同的Signers(这里应该理解为为image签发证书的人)拥有不同的信任级别，这导致了不同等级的保护进程之间受保护的程度不同

同时PPL可以让Signers定义哪些访问权限不能被较低保护等级的进程获取，例如可以拒绝较低保护等级的进程获取保护进程的PROCESS_TERMINATE权限

Internal Protection Process Level Symbol Level Protection Type Signer
PS_PROTECTED_SYSTEM 0x72 Protected WinSystem
PS_PROTECTED_WINTCB 0x62 Protected WinTcb
PS_PROTECTED_WINTCB_LIGHT 0x61 Protected Light WinTcb
PS_PROTECTED_WINDOWS 0x52 Protected Windows
PS_PROTECTED_WINDOWS_LIGHT 0x51 Protected Light Windows
PS_PROTECTED_LSA_LIGHT 0x41 Protected Light Lsa
PS_PROTECTED_ANTIMALWARE_LIGHT 0x31 Protected Light Anti-malware
PS_PROTECTED_AUTHENTICODE 0x21 Protected Authenticode
PS_PROTECTED_AUTHENTICODE_LIGHT 0x11 Protected Light Authenticode
PS_PROTECTED_NONE 0x00 None None
  • 同一个Singer签署的进程中，Protected > Protected Light
  • 高级别的保护进程可以访问低级别的保护进程

一个镜像文件的保护等级不仅取决于签发者，还取决于证书中EKU的值

一个受保护的进程加载DLL时，该DLL也会收到相应的审查，只有具备适当签名的DLL才能被加载进保护进程的地址空间中

一些Windows关键进程会被要求以某一最低保护等级来启动运行，详见part1 p118

Third-party PPL support

除了微软官方签署的系统组件外，PPL最常见的用途是用于杀毒软件(反恶意软件软件 anti-malware software)，杀软常由三部分组成

  • 驱动 拦截文件系统或网络的I/O请求，实现拦截功能
  • 用户模式服务 配置驱动、与驱动进行通信，接受驱动下发的事件
  • 用户模式GUI进程

用户模式服务进程需要以PPL运行来保护自身，杀软驱动内可以存放一些Singer以及EKU信息，当代码完整性系统检查到某个文件具有这些信息的时候，它就会允许该进程以PPL运行

Minimal and Pico process

并非所有的进程都是为了执行用户态的代码；例如System process仅用作绝大多数系统线程以及驱动句柄的容器

Minimal processes

向NtCreateProcessEx传递特殊的参数即可创建minimal process，这样的进程不具有许多正常进程独有的结构

  • 不会分配用户模式地址空间，因此也没有PEB
  • NTDLL不会被映射到进程中，也不会有任何加载器/API Set信息
  • 没有section对象与进程绑定，这意味着没有可执行文件镜像与该进程关联
  • EPROCESS会设置特殊的标志，这会导致所有的线程也成为minimal thread

Pico processes

Pico provider必须在所有其他第三方驱动之前加载，而且只有少数十余个核心驱动允许调用PsRegisterPicoProvider

在注册Pico Provider时，内核会向Pico Provider提供一系列函数用于管理Pico process，同时Pico Provider也会向内核提交一系列函数用作接管Pico Process触发的一系列事件

Pico Provider本质上是一个实现了一系列重要回调函数的内核模块，这些回调函数能够响应Pico process触发的一系列事件

Trustlets (secure processes) (本章未仔细阅读)

trustlets运行在VTL 1中，VTL 1的ring 3进程也可以避免来自VTL 0的ring 0及ring 3代码的访问

trustlets只能导入特定的若干DLL，它们能使用的系统调用也有限制，同时它们的镜像文件中会有一个特殊的.tPolicy段，里面是一些关于如何处理来自VTL 0访问的设定
